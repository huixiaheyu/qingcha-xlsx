# 清查表字段映射清单

> 目标：把“固定模板 + 多个输入表 + 按需填充 + 没有来源则为空”的规则写清楚。后续检查某一列是否填错时，优先看这份清单，再看代码和测试。

## 1. 总原则

### 1.1 固定模板原则

当前系统只服务这份固定模板：

- 模板文件：`完整/清查表.xlsx`
- 输出文件：`完整/清查表（已填充）.xlsx`

模板决定：

- 目标工作表名称
- 表头行数
- 数据起始行
- 模板行
- 目标列含义
- 哪些列必须保留为空
- 哪些单元格不能被覆盖，例如 `组织账套表!F4` 的 Excel 日期单元格

### 1.2 有来源才填，无来源为空

所有目标字段按以下优先级处理：

1. 有明确源字段：直接填。
2. 有明确转换规则：转换后填。
3. 有确定性匹配规则：匹配成功后填。
4. 没有来源、来源不确定、匹配不唯一：留空。

禁止：

- 猜值
- 用相邻列代替
- 复制模板旧值
- 把表头文本留到数据区
- 为了“看起来不空”而写死占位值

### 1.3 三层映射

当前代码按三层处理：

```text
源 Excel 列字母 -> loader 转成语义字段 -> filler 写入目标模板列
```

这样做的目的：

- 源表列变化时，优先检查 `load_*()`。
- 目标模板列变化时，优先检查 `fill_*_sheet()`。
- 跨表规则变化时，优先检查 helper 或匹配函数。

## 15. 当前网页化可行性评估

### 15.1 能不能做成网页？

能做，但要分两种交付方式。

### 15.2 方式一：直接交付一个单独 HTML 文件

结论：技术上可行，但不建议作为第一版正式交付。

可行原因：

- 浏览器可以读取用户上传的 `.xlsx` 文件。
- 可以用前端 JS 解析和生成 Excel，例如使用 SheetJS 这类库。
- 用户可以在网页里上传模板和输入表，然后下载填好的 `清查表（已填充）.xlsx`。

主要问题：

1. 当前 Python 逻辑依赖直接编辑 XLSX 内部 XML。
   - 它会克隆模板行。
   - 它会保留多行表头。
   - 它会保留特定单元格格式，例如日期单元格。
   - 这些在纯前端重写时需要重新实现和验证。

2. 单 HTML 文件如果要离线使用，依赖库也要内嵌。
   - 文件会变大。
   - 后续维护不如 Python 脚本直接。

3. Excel 样式、合并单元格、模板行克隆是高风险点。
   - 如果前端库写回时破坏样式，用户肉眼会看到格式变化。

适合场景：

- 给非技术用户一个本地网页工具。
- 用户只需拖入文件、点按钮、下载结果。
- 数据不出浏览器，隐私较好。

不适合场景：

- 当前阶段快速稳定交付。
- 仍在频繁核对字段映射。
- 非常依赖模板格式完全不变。

### 15.3 方式二：网页前端 + Python 后端

结论：最稳，适合作为正式工具。

形态：

```text
网页上传文件 -> Python 后端调用现有 build_qingcha_table 逻辑 -> 返回填充后的 xlsx
```

优点：

- 可以复用当前已经验证过的 Python 逻辑。
- 17 条测试仍然有效。
- Excel XML 写入行为不需要重写。
- 后续只维护一套核心逻辑。

缺点：

- 需要运行一个本地服务或服务器。
- 不是一个单独 HTML 文件。

适合交付方式：

- 本地小工具：双击启动一个 Python 程序，然后浏览器打开页面。
- 内网工具：部署在一台电脑或服务器上，大家通过浏览器使用。

### 15.4 推荐路线

我建议分两步：

#### 第一步：先做“本地网页 + Python 后端”

原因：

- 最大程度复用当前脚本。
- 风险低。
- 很快能做出用户界面。
- 仍然保证模板格式和当前测试行为。

界面可以很简单：

```text
上传/选择：
- 用户信息表.xlsx
- 科目期初数据明细.xlsx
- 科目余额表数据.xlsx
- 固定资产统计数据.xlsx
- 水井固定资产统计数据.xlsx
- 资源明细数据.xlsx
- 合同明细数据.xlsx

不需要上传：
- 清查表.xlsx（服务端固定模板）

按钮：生成清查表（已填充）.xlsx

日志：显示已接收文件、忽略文件、缺失文件、未确定匹配资源、生成成功/失败

结果：下载生成文件
```

#### 第二步：等映射稳定后，再考虑单 HTML 离线版

如果要单 HTML 版，需要重新实现：

- xlsx 读取
- sheet 定位
- 模板行克隆
- 单元格样式保留
- 多行表头保留
- 输出 xlsx 下载

这一步适合在当前 Python 逻辑完全稳定后做。

### 15.5 最终判断

| 方案 | 是否可行 | 推荐度 | 原因 |
|---|---|---:|---|
| 继续 Python 脚本 | 可行 | 高 | 当前已验证，最快最稳 |
| 本地网页 + Python 后端 | 可行 | 最高 | 有界面，同时复用现有逻辑 |
| 单独 HTML 文件 | 可行但风险高 | 中低 | 需要重写 Excel 处理逻辑，样式风险大 |

### 15.6 当前已实现的前后端形态

当前实现采用单个 FastAPI 服务同时提供 API 和静态页面，不使用 Nginx：

```text
浏览器页面 -> FastAPI /api/build -> 临时上传目录 -> 服务端固定模板 -> 下载生成文件
```

部署入口：

```text
docker compose up --build
http://localhost:8080
```

接口约定：

| 接口 | 用途 |
|---|---|
| `GET /api/health` | 健康检查 |
| `GET /api/required-files` | 返回 7 个必需源文件名，并声明模板由服务端提供 |
| `POST /api/build` | 接收多文件上传，生成 `清查表（已填充）.xlsx` |
| `GET /api/download/{job_id}` | 下载指定任务生成的工作簿 |

上传规则：

- 用户只上传 7 个源工作簿，不上传 `清查表.xlsx`。
- 如果上传了 `清查表.xlsx`，后端会忽略，并在日志中说明使用服务端模板。
- 如果上传了非必需文件，后端会忽略并写日志。
- 每次生成使用独立临时目录，源文件和输出文件不会写回 `完整/`。
- 缺失必需文件时返回 400，并在日志和 `missingFiles` 中列出缺失项。
- 构建失败时返回 422，并把失败原因写入日志。

建议结论：

> 当前行为可以做成网页，但第一版不建议直接交付单 HTML 文件。已实现的交付方式是“静态网页界面 + FastAPI 后端 + Docker Compose”，不使用 Nginx，模板由服务端固定提供，用户只上传 7 个源工作簿。
